# GRP-003 — IDs carried forward across pipeline stages

> Category **GRP** (graph integrity) · Scope **project** · Default severity **warning** · Fixable **no**
> · [Rules index](README.md) · [Configuration](../configuration.md)

## What it checks

When documents form a **traceability pipeline** — requirements flow into design, design into
tests, and so on — every ID declared at one stage should be picked up at the next. GRP-003 walks
a declared `chain` of stages and reports any ID that appears in a stage's `idColumn` but is not
referenced in the following stage's `refColumn`.

For each adjacent pair of stages, the rule collects the IDs referenced at the next stage
(`refColumn`), then checks that every ID defined at the current stage (`idColumn`) appears among
them. An ID matched at stage *N* that never shows up at stage *N+1* is flagged as dropped.

Unlike GRP-001 and GRP-002, this rule is **graph-independent**: it reads the declared table
columns directly rather than the [ContextGraph](../context-graph.md). Its scope is still
`project` because a chain spans multiple documents. A stage without an `idColumn` defines no IDs
of its own, so it is only checked as a *reference* target for the previous stage, never as a
source.

## Options

| Option | Type | Required | Default | Description |
| --- | --- | --- | --- | --- |
| `chain` | `Stage[]` (≥2) | **yes** | — | Ordered list of pipeline stages; at least two are required. |
| `idPattern` | `string` (regex) | no | `^.+$` | Only IDs matching this pattern are extracted and tracked. |

Each **`Stage`** in `chain`:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `stage` | `string` (non-empty) | **yes** | Human-readable stage name used in the finding message. |
| `files` | `string[]` (≥1) | **yes** | Glob(s) selecting the documents that make up this stage. |
| `idColumn` | `string` (non-empty) | no | Table column that *defines* IDs at this stage. A stage with no `idColumn` defines nothing to carry forward. |
| `refColumn` | `string` (non-empty) | **yes** | Table column that *references* the previous stage's IDs. |

## Example

### Config

```jsonc
{
  "rules": [
    {
      "rule": "GRP-003",
      "options": {
        "idPattern": "^REQ-\\d+$",
        "chain": [
          { "stage": "requirements", "files": ["reqs.md"], "idColumn": "ID", "refColumn": "ID" },
          { "stage": "design", "files": ["design.md"], "refColumn": "Requirement" }
        ]
      }
    }
  ]
}
```

### Passes

```md
<!-- reqs.md -->
| ID |
| --- |
| REQ-1 |

<!-- design.md -->
| Requirement |
| --- |
| REQ-1 |
```

Every requirement ID (`REQ-1`) is referenced in the design stage's `Requirement` column.

### Fails

```md
<!-- reqs.md -->
| ID |
| --- |
| REQ-1 |
| REQ-2 |

<!-- design.md -->
| Requirement |
| --- |
| REQ-1 |
```

→ GRP-003 reports **`ID "REQ-2" from stage "requirements" is not carried into stage "design".`**,
anchored at the `REQ-2` row in `reqs.md`. `REQ-1` is fine; `REQ-2` was dropped.

## Notes

- **Scope:** `project` — a chain spans multiple documents, but the check reads declared columns,
  not the ContextGraph.
- **`chain` needs at least two stages** — a pipeline with one stage has nothing to carry an ID
  *into*.
- **`idPattern`** filters both defined and referenced IDs before comparison; the default `^.+$`
  accepts any non-empty cell.
- A stage that omits `idColumn` contributes no IDs of its own; it only serves as the reference
  target (`refColumn`) for the stage before it.
