# CTX-003 — Use canonical glossary terms instead of aliases

> Category **CTX** (content) · Scope **project** · Default severity **warning** · Fixable **no**
> · [Rules index](README.md) · [Configuration](../configuration.md)

## What it checks

Content should refer to a concept by its **canonical term** rather than by a known alias. CTX-003
reads a glossary table, builds an alias → canonical-term map from it, then scans the other in-scope
files for alias usage and reports each occurrence.

This keeps a project's vocabulary consistent: if the glossary declares `GraphQL` as canonical with
`gql` and `graphql` as aliases, prose that writes `gql` is flagged with a suggestion to use
`GraphQL`. Because it correlates a glossary file against the rest of the repository, its scope is
**project** (it sees every parsed document at once, not one file in isolation).

Matching is **whole-word**: an alias `api` matches the word `api` but not `APIs` or `myapi`. The
glossary file itself is never flagged. An alias identical to the canonical term is ignored.

### How the glossary is read

- `glossary` is a glob selecting the file(s) that hold the glossary table.
- In each glossary table, CTX-003 looks for a header named by `termColumn`; tables without that
  header are skipped. The value in that column is the **canonical term**.
- Aliases come from the `aliasColumn` cell, split on commas. If `aliasColumn` is omitted, no
  aliases are derived and the rule has nothing to enforce.

## Options

| Option | Type | Required | Default | Description |
| --- | --- | --- | --- | --- |
| `glossary` | `string` (≥1 char) | **yes** | — | Glob selecting the glossary file(s) to read canonical terms and aliases from. |
| `termColumn` | `string` (≥1 char) | **yes** | — | Header name of the column holding the canonical term. |
| `aliasColumn` | `string` (≥1 char) | no | — | Header name of the column holding comma-separated aliases; without it, no aliases are enforced. |
| `section` | `string` | no | — | Only scan content under a heading matching this text; omit to scan whole files. |
| `files` | `string[]` | no | all included | Glob(s) narrowing which files this rule instance scans (the glossary is always excluded from scanning). |
| `exclude` | `string[]` | no | — | Glob(s) removing files from the scan. |

## Example

### Config

```jsonc
{
  "rules": [
    {
      "rule": "CTX-003",
      "options": {
        "glossary": "docs/glossary.md",
        "termColumn": "Term",
        "aliasColumn": "Aliases",
        "files": ["docs/**/*.md"]
      }
    }
  ]
}
```

### Glossary file (`docs/glossary.md`)

```md
| Term    | Aliases       |
| ------- | ------------- |
| GraphQL | graphql, gql  |
```

### Fails

```md
We expose a gql endpoint and a graphql schema.
```

→ CTX-003 reports two findings — one for **`gql`** and one for **`graphql`** — each suggesting the
canonical term **`GraphQL`**, anchored at the line where the alias appears.

### Scoped to one section

```jsonc
{
  "rule": "CTX-003",
  "options": { "glossary": "docs/glossary.md", "termColumn": "Term", "aliasColumn": "Aliases", "section": "Overview" }
}
```

Only content beneath an `Overview` heading is scanned for alias usage.

## Notes

- **Scope:** `project` — it reads the glossary and the scanned files together, not one file in
  isolation.
- The glossary file is always excluded from scanning, so its own alias cells are never flagged.
- Alias matching is whole-word; substrings inside larger identifiers do not match.
- With no `aliasColumn`, or when the glossary yields no aliases, the rule reports nothing.
