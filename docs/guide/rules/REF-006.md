# REF-006 — References do not depend on less-stable entities

> Category **REF** (references) · Scope **project** · Default severity **warning** · Fixable **no**
> · [Rules index](README.md) · [Configuration](../configuration.md)

## What it checks

REF-006 enforces a stability ordering across dependencies: an entry must not depend on something
*less stable* than itself. This catches, for example, a `stable` API that references an
`experimental` one — a leak of instability that consumers would not expect.

You give the rule a stability vocabulary via `stabilityOrder`, listed from **least stable to most
stable** (its position is the rank). Definition files carry, per entity, an ID (`idColumn`) and a
stability (`stabilityColumn`). Reference files carry rows where the `idColumn` lists the IDs the
row depends on and the `stabilityColumn` holds the *referencer's own* stability.

For each reference row, REF-006 looks up the stability of each referenced ID from the definition
tables and compares ranks. If a referenced entity is less stable (lower rank) than the referencing
row, it reports a finding. Rows whose stability value is not in `stabilityOrder` are skipped, as
are referenced IDs with no known stability.

This is a **local-only** check computed from table contents in the corpus; it performs no network
access.

## Options

| Option | Type | Required | Default | Description |
| --- | --- | --- | --- | --- |
| `stabilityColumn` | `string` (non-empty) | **yes** | — | Column header holding an entity's (and a referencer's) stability level. |
| `stabilityOrder` | `string[]` (≥2) | **yes** | — | Stability values from least to most stable; index is the rank. |
| `definitions` | `string[]` (≥1) | **yes** | — | Glob(s) selecting files whose tables define entity stability. |
| `references` | `string[]` (≥1) | **yes** | — | Glob(s) selecting files whose tables declare dependencies. |
| `idColumn` | `string` (non-empty) | **yes** | — | Column header carrying the entity ID (in definitions) or the referenced IDs (in references). |
| `idPattern` | `string` (regex) | no | `^.+$` | Regex a token in a reference cell must match to count as a referenced ID. |

## Example

### Config

```jsonc
{
  "rules": [
    {
      "rule": "REF-006",
      "options": {
        "stabilityColumn": "Stability",
        "stabilityOrder": ["experimental", "stable"],
        "definitions": ["defs.md"],
        "references": ["refs.md"],
        "idColumn": "ID"
      }
    }
  ]
}
```

### Fails

`defs.md` (definitions):

```md
| ID | Stability    |
| -- | ------------ |
| A  | stable       |
| B  | experimental |
```

`refs.md` (references):

```md
| ID | Stability |
| -- | --------- |
| B  | stable    |
```

→ REF-006 reports (as a **warning**) that the row **depends on `B` (stability `experimental`)
which is less stable than this entry (stability `stable`)**, anchored at the reference row.

### Passes

A `stable` entry that depends only on other `stable` (or more stable) entities produces no
finding, because no referenced entity has a lower rank than the referencer.

## Notes

- **Scope:** `project` — evaluated over the whole corpus so definitions and references can live in
  different files.
- The `idColumn` does double duty: in definition tables it names the entity's own ID; in reference
  tables it lists the IDs the row depends on. Multiple referenced IDs may appear in one cell,
  separated by whitespace or commas.
- Comparison is by rank in `stabilityOrder`; values absent from that list (on either side) are
  ignored rather than treated as errors.
