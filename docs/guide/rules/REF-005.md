# REF-005 — IDs are traceable between definitions and references

> Category **REF** (references) · Scope **project** · Default severity **error** · Fixable **no**
> · [Rules index](README.md) · [Configuration](../configuration.md)

## What it checks

REF-005 enforces two-way traceability of identifiers across your corpus:

- **Dangling reference (error):** an ID used in a *reference* file has no matching definition.
- **Orphan definition (warning):** an ID declared in a *definition* file is never referenced.

You tell the rule which files hold definitions (`definitions`), which hold references
(`references`), which table column carries the ID (`idColumn`), and which token shape counts as an
ID (`idPattern`). REF-005 then builds the set of defined IDs and the set of referenced IDs across
the whole project and compares them.

Definitions are discovered through the shared ID-extraction logic, which looks at both the ID
column and headings, so REF-005 agrees with the context graph's `id-ref` edges on what counts as
a definition. References are column-only: they are read from the `idColumn` of the reference
files. Both severities are emitted from a single project-scope pass; the severity is fixed per
finding (dangling → error, orphan → warning) regardless of the rule's configured severity.

## Options

| Option | Type | Required | Default | Description |
| --- | --- | --- | --- | --- |
| `definitions` | `string[]` (≥1) | **yes** | — | Glob(s) selecting files that *define* IDs. |
| `references` | `string[]` (≥1) | **yes** | — | Glob(s) selecting files that *reference* IDs. |
| `idColumn` | `string` (non-empty) | **yes** | — | Table column header that carries the ID in both definition and reference tables. |
| `idPattern` | `string` (regex) | **yes** | — | Regular expression a cell value must match to count as an ID. |

## Example

### Config

```jsonc
{
  "rules": [
    {
      "rule": "REF-005",
      "options": {
        "definitions": ["reqs.md"],
        "references": ["design.md"],
        "idColumn": "ID",
        "idPattern": "^REQ-\\d+$"
      }
    }
  ]
}
```

### Passes / Fails

`reqs.md` (definitions):

```md
| ID    |
| ----- |
| REQ-1 |
| REQ-2 |
```

`design.md` (references):

```md
| ID    |
| ----- |
| REQ-1 |
| REQ-9 |
```

→ REF-005 reports:
> - **error** in `design.md`: reference **`REQ-9` has no definition** (dangling).
> - **warning** in `reqs.md`: definition **`REQ-2` is never referenced** (orphan).
>
> `REQ-1` is both defined and referenced, so it is fine.

## Notes

- **Scope:** `project` — evaluated once over the whole corpus rather than per file, so
  definitions and references in different files are matched.
- **`settings.idRef` and the graph:** ID references also feed the shared context graph's `id-ref`
  edges via `settings.idRef` (`{ idPattern, definitions, idColumn }`), which count toward
  [GRP-001](GRP-001.md) cycles and [GRP-002](GRP-002.md) incoming references. REF-005 does **not**
  expose its resolved options to the graph builder, so a project that wants both ID traceability
  here and ID-aware graph analysis must configure the same shape in **both** `settings.idRef` and
  this rule's options. See [Configuration](../configuration.md).
- `idPattern` is applied per cell value; tokens that do not match are ignored, which keeps
  free-text cells from being treated as IDs.
