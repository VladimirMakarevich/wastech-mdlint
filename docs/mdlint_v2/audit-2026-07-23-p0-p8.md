### 🟠 MEDIUM

#### M-1. Неверная строка/колонка у импортов в многострочных блоках `@import`
- **Категория:** баг позиционирования (перепроверено вручную).
- **Где:** `packages/core/src/markdown/parse-document.ts:209-232` (`extractImports`).
- **Суть:** remark упаковывает несколько подряд идущих непустых строк в **один** узел
  `text`. Функция берёт `line: node.position.start.line` для **каждого** совпадения и
  считает `column` как байтовое смещение от начала узла. В результате 2-й, 3-й и т.д.
  импорты в блоке вида
  ```
  @AGENTS.md
  @.agents/rules/architecture.md
  @.agents/rules/coding-style.md
  ```
  (ровно паттерн из `CLAUDE.md` этого же репозитория) все репортятся как `line: 1` с
  фиктивной большой колонкой вместо строк 2 и 3.
- **Почему важно:** импорты питают бюджет «eager import» (D3) и рёбра графа типа
  `import` (P4), которые несут `line` для объяснимости (G3). Набор `rawTarget` при этом
  полный — портятся только позиции. Существующие тесты покрывают лишь однострочные
  импорты (`parse-document.test.ts:145-159`), поэтому баг не пойман.
- **Рекомендация:** вычислять строку/колонку от абсолютного offset совпадения
  внутри `node.value` с учётом переносов строк, а не от начала узла.

#### M-2. `custom` target `"heading"` заявлен в требованиях/глоссарии, но не реализован
- **Категория:** контракт over-promise (перепроверено).
- **Где:** `requirements/02-rules-engine.md:45` и `glossary.md:262,265` перечисляют
  набор `table | section | content | checklist | link | heading`. Но
  `ASSERTION_TARGETS` (`engine/primitives/assert.ts:85`) и enum сгенерированной схемы
  (`engine/schema.ts:85,101` → `["checklist","content","link","section","table"]`)
  **не содержат `heading`**, и ни один assertion-примитив не работает с заголовками.
- **Почему важно:** пользователь, следующий каноническому глоссарию и написавший
  `target: "heading"`, получит отказ на этапе валидации схемы. Глоссарий объявлен
  каноническим источником — здесь он обещает несуществующую поверхность.
- **Рекомендация:** либо убрать `heading` из требований/глоссария, либо добавить
  примитив, работающий с заголовками.

#### M-3. MCP-инструмент `lint` в описании утверждает, что не читает ФС
- **Категория:** нечестное описание инструмента (перепроверено). Нарушает M2 из
  `requirements/05-mcp-server.md` (описания должны соответствовать реальному поведению).
- **Где:** `packages/mcp-server/src/tools/lint.ts:177` — описание *«Reads no filesystem
  or config.»*. Но `lint.ts:141` задаёт `rootDir: process.cwd()`, а REF-001/REF-003
  резолвят относительные ссылки/картинки через `existsSync(path.resolve(rootDir, …))`
  (`engine/primitives/reference.ts:23,109`).
- **Почему важно:** вызов `lint` с REF-правилом и относительной ссылкой заставит
  инструмент делать `stat` путей под cwd сервера — прямое противоречие описанию.
- **Рекомендация:** уточнить копирайтинг, напр. «конфиг не читает; для reference-правил
  может проверять существование целей относительно cwd».

#### M-4. `loadDocuments` сортирует через `localeCompare` — риск недетерминизма между окружениями
- **Категория:** слабое место (детерминизм/кросс-платформенность).
- **Где:** `packages/core/src/markdown/load-documents.ts:171` —
  `results.sort((a, b) => a.localeCompare(b))`. Именно эта сортировка фиксирует
  порядок Map (пользовательский контракт детерминизма, P1.05).
- **Почему важно:** `localeCompare` без явной локали/опций зависит от реализации и
  ICU-локали среды; порядок для смешанного регистра, пунктуации и CJK/Unicode
  (которые проект явно поддерживает) может отличаться между машинами. Тест
  детерминизма (`load-documents.test.ts:97-108`) прогоняет на одной машине и это не
  ловит.
- **Рекомендация:** сравнение по кодовым точкам (`<`/`>`) либо фиксированная локаль.

#### M-5. Нет кросс-ОС матрицы в CI — нормализация путей под Windows заявлена, но не проверяется
- **Категория:** слабое место (пробел верификации).
- **Где:** `.github/workflows/ci.yml:16,35` — оба job `runs-on: ubuntu-latest`; матрица
  есть только по пакетам для `npm pack`. Windows-нормализация (`load-documents.ts:24`
  `toPosixAbsolute`) проверяется лишь `load-documents.test.ts:45` (`!key.includes("\\")`),
  что на POSIX-хосте тривиально истинно.
- **Почему важно:** критерий выхода P1 «Windows-пути нормализованы в POSIX» и
  архитектурный инвариант «корректно на Windows/macOS/Linux» фактически не
  верифицируются. Регрессии путей/glob/переводов строк на Windows уедут в релиз
  незамеченными. Сам код нормализации выглядит корректным — пробел именно в проверке.
- **Рекомендация:** добавить `windows-latest` (и по возможности `macos-latest`) в
  матрицу verify-job.

#### M-6. Гейт форматирования красный на 203 файлах и не запускается в CI
- **Категория:** слабое место (инфраструктура/дисциплина).
- **Где:** `npm run format` (`prettier --check .`) сообщает «Code style issues found in
  203 files». При этом `.github/workflows/ci.yml` в job `verify` гоняет typecheck /
  lint / test / build, но **не** `format`.
- **Почему важно:** `npm run format` документирован как гейт в `AGENTS.md` и
  `.agents/rules/testing.md`, но по факту красный по всему репозиторию и никем не
  контролируется. Либо гейт нужно чинить и включать в CI, либо честно понизить его
  статус в документации.
- **Рекомендация:** прогнать `prettier --write`, затем добавить `npm run format` в CI
  (или зафиксировать сознательный отказ от гейта в правилах).

#### M-7. Governance-доки утверждают о наличии корневых `src/` и `test/`, которых уже нет
- **Категория:** противоречие (документация ↔ реальность).
- **Где:** `AGENTS.md:11` («…still contains the single-package codebase in `src/` and
  `test/`») и `.agents/rules/architecture.md:17` («…still contains … code in `src/`»).
  Фактически `src/` и `test/` в корне отсутствуют; `packages/core/src/index.ts:5`
  фиксирует, что легаси-пайплайн «был удалён на cutover P3.09».
- **Почему важно:** эти же файлы предписывают «считать текущее состояние ФС истиной о
  расположении кода» — то есть документ одновременно утверждает удалённую структуру.
  Это самопротиворечие дезориентирует агентов и людей относительно того, где лежит код.
- **Рекомендация:** обновить формулировки под пост-P3.09 состояние (код целиком в
  `packages/*`).

#### M-8. Маркеры статуса фаз в глоссарии устарели
- **Категория:** противоречие (документация ↔ реальность).
- **Где:** `glossary.md:25-27,632-635` («P0–P5 shipped; P6–P9 not started»);

- **Единый обход графа:** `slice`/`impact`/compile-budget идут через `query()`; параллельных
  реализаций обхода для потребителей нет.

  контракт ошибок `{code,message,hint}` с санитизацией `INTERNAL_ERROR`; инвентарь

  `$ARGUMENTS`, нет bang-инъекций; плейсхолдеры заменены), enforced тестом.
- **init:** нет install-time записей (нет `postinstall`), Ctrl+C → exit 0, JSONC-writer с
  локальным `$schema`.
- **Конфиг:** JSONC, локальный `$schema`, нет `.cjs`/`.mjs`/runtime-TS загрузки, нет
  удалённых schema-URL (единственный `https://` в `schema.ts:15` — это диалект
  JSON Schema, стандарт).
- **Docs/schema sync:** README (таблица правил + инвентарь MCP) и `packages/cli/schema.json`
  совпадают с генератором; sync-тесты зелёные.
- **README ↔ CLI:** все команды и флаги README совпадают с `program.ts`.
- **Зависимости:** три пакета на `0.0.0`, внутренние deps `"0.0.0"` (не `workspace:*`, по
  D1), `engines.node >= 24.17.0` единообразно.

   многострочный `@import`.

   требований/глоссария (**M-2**).

   гейта в правилах (**M-6**).

   глоссарии (**M-8**).
